PROMPT ENGINEERING
Domain 1 | Task 1.6 | Estimated: 35 minutes

================================================================================
WHY IT MATTERS
================================================================================
Here's the best-kept secret in AI: before you fine-tune, before you switch models, before you add complexity—fix your prompts. A well-crafted prompt can transform mediocre outputs into impressive ones at zero additional cost. This is often the highest-leverage optimization available. Zero-shot, few-shot, chain-of-thought, and prompt governance are techniques that work across every GenAI use case.

================================================================================
CONCEPT: Core Prompting Techniques
================================================================================
The way you ask matters as much as what you ask. Small changes to prompt structure can dramatically change output quality. Understanding these techniques is foundational.

**Zero-shot prompting** gives instructions without examples. You describe what you want; the model uses its training to figure it out. 'Summarize this article in three bullet points'—the model knows what summarization means. This works great for straightforward tasks where the model already understands the domain.

**Few-shot prompting** shows 2-5 examples before your actual request. Instead of explaining exactly how you want customer support responses formatted, show three excellent examples. The model learns the pattern through demonstration:

Example 1:
Customer: My order hasn't arrived.
Response: I understand how frustrating delayed orders can be. Let me look up your order #[ORDER_ID] and provide a status update.

Example 2:
Customer: I want a refund.
Response: I'd be happy to help with your refund. Could you share your order number so I can review the details?

Now respond to:
Customer: {{user_message}}

Few-shot is powerful when the format or style is hard to describe in words. **Pro tip**: Include both positive AND negative examples—show what NOT to do.

**Chain-of-thought (CoT)** asks the model to reason step-by-step. This is a game-changer for complex reasoning—math problems, logic puzzles, multi-step analysis. Just adding 'Let's think through this step by step' before asking for an answer often **doubles accuracy**:

A store has 150 apples. They sell 40% on Monday and 30 more on Tuesday. How many remain?

Let's solve this step by step:
1. Monday sales: 150 x 0.40 = 60 apples
2. After Monday: 150 - 60 = 90 apples
3. Tuesday sales: 30 apples
4. Final count: 90 - 30 = 60 apples

**Structured output** requests specific formats like JSON or XML. Instead of 'Extract the key information,' specify 'Extract the following fields in JSON format: {name, date, amount}.' This makes outputs parseable by downstream systems. Essential for integrating AI into larger workflows.

**Self-consistency** runs the same prompt multiple times and picks the most common answer—useful for high-stakes decisions where you need confidence.

**Role prompting** sets the model's persona: 'You are an expert AWS solutions architect.' This primes domain knowledge and response style.

These techniques combine naturally. Use few-shot to establish format, chain-of-thought for reasoning, role prompting for expertise, and structured output for parseability—all in one prompt.

KEY POINTS:
- Zero-shot: Direct instruction, relies on model's training knowledge
- Few-shot: 2-5 examples including what NOT to do
- Chain-of-thought: 'Think step by step' can DOUBLE accuracy on reasoning
- Role prompting: Set persona to prime domain expertise

EXAM TIP: 'Reasoning' or 'complex logic' → chain-of-thought. 'Consistent format' → few-shot. 'Domain expertise' → role prompting.

CONCEPT: Prompt Management and Governance
================================================================================
Production AI needs more than prompts scattered in code. As complexity grows and teams iterate, you need systematic management—versioning, testing, approval workflows. Otherwise you get chaos.

**Bedrock Prompt Management** is your single source of truth. Store prompts centrally with version control. Every change is tracked—who modified what, when. Roll back instantly if quality degrades. This beats prompts buried in application code where changes are hard to coordinate.

**Parameterized templates** separate structure from content. Define templates with placeholders: 'Analyze the following {{document_type}} and extract {{required_fields}}.' At runtime, fill in the blanks. This maintains quality while allowing flexibility—changing document type doesn't risk breaking the prompt structure.

**Approval workflows** ensure prompts meet standards before reaching production. In regulated industries or high-stakes apps, prompt changes should go through review. Prompt Management supports IAM access controls. Build approval workflows with Step Functions for formal review processes.

**Testing is essential**. Create evaluation datasets covering expected inputs and edge cases. Run prompts against test sets before deployment. Automated testing catches quality regressions when prompts change. Use Lambda and Step Functions to build testing pipelines.

**CloudTrail** logs prompt access and modifications for compliance. Who accessed this prompt? What changed? When? Essential for regulated industries and debugging production issues.

KEY POINTS:
- Version control: Track prompt changes with immutable versions
- Parameterized templates: {{variable}} placeholders for dynamic content
- Testing: Evaluate prompts against test datasets before production
- Approval workflows: Review process for prompt changes

EXAM TIP: Bedrock Prompt Management = 'version control,' 'templates,' 'governance,' or 'centralized prompt storage.'

CONCEPT: Prompt Flows and Chaining
================================================================================
Complex tasks often need multiple model calls, each building on the last. Prompt chaining breaks problems into stages—the output of one prompt feeds the next.

Consider document analysis: first summarize, then extract entities from the summary, then generate recommendations based on those entities. Each stage uses a focused prompt. This often beats a monolithic prompt trying to do everything at once.

**Bedrock Prompt Flows** provides visual, no-code orchestration. Design flows graphically, connecting prompts with data transformations and conditional logic. Perfect for business users or rapid prototyping—build and test pipelines without writing code.

**Step Functions** offers more power for complex logic. When you need sophisticated branching, parallel execution, error handling with retries, or deep AWS integration, Step Functions delivers the control that visual tools can't match.

**Conditional branching** routes to different prompts based on model output. A classification stage determines input type—question, complaint, feedback—then routes to specialized prompts for each. This enables sophisticated, adaptive AI systems.

**Reusable components** prevent duplication. Extract common patterns—standard system messages, output format instructions—into shared components. Improve a component once, all flows using it benefit. Consistency and maintainability in one package.

KEY POINTS:
- Prompt Flows: Visual, no-code orchestration in Bedrock console
- Step Functions: Complex branching, retries, and error handling
- Prompt chaining: Output of one prompt feeds into the next
- Conditional logic: Route to different prompts based on FM output

CONCEPT: Output Quality Assurance
================================================================================
Foundation model outputs are probabilistic. The same prompt can produce different outputs. Some might be wrong, harmful, or malformed. Production systems must validate before delivering to users or downstream systems.

**Structured output validation** catches format errors. If you requested JSON, parse it and verify the schema. Missing fields? Invalid values? Either retry with clarification or return an error. Don't pass malformed data downstream where it causes harder-to-debug failures.

**Bedrock Guardrails** provides built-in content filtering with multiple policy types:
- Content filters: Block harmful content (hate, violence, sexual, misconduct) with configurable strength levels (LOW/MEDIUM/HIGH)
- Denied topics: Define specific topics the model should refuse to discuss
- Word filters: Block specific words or phrases
- PII filters: Detect and mask/block sensitive data (SSN, credit cards, emails, names, addresses)
- Contextual grounding: Check if responses are grounded in provided context (reduces hallucination)

Apply guardrails to both **inputs** (preventing prompt injection) AND **outputs** (filtering inappropriate responses):

// Check input before sending to model
const inputCheck = await client.send(new ApplyGuardrailCommand({
  guardrailIdentifier: 'my-guardrail',
  guardrailVersion: 'DRAFT',
  source: 'INPUT',
  content: [{ text: { text: userMessage } }]
}));

if (inputCheck.action === 'GUARDRAIL_INTERVENED') {
  return { error: 'Request blocked by content policy' };
}

// Invoke model, then check output
const outputCheck = await client.send(new ApplyGuardrailCommand({
  guardrailIdentifier: 'my-guardrail',
  guardrailVersion: 'DRAFT',
  source: 'OUTPUT',
  content: [{ text: { text: modelResponse } }]
}));

**Lambda post-processing** implements custom business rules. 'Never recommend competitor products.' 'Prices must be within valid ranges.' Business-specific logic that guardrails can't express. Lambda validates, transforms, or enriches FM outputs before they reach users.

**Quality monitoring** detects degradation over time. Log outputs with quality metrics. Build CloudWatch dashboards tracking response quality, guardrail trigger rates, and validation failures. Set alarms when metrics deviate from baselines. Model behavior can drift subtly—catching problems early prevents user impact.

KEY POINTS:
- Guardrails policies: Content filters, denied topics, word filters, PII, grounding
- PII handling: Detect and mask/block sensitive data automatically
- Apply to inputs AND outputs: Prevent injection, filter inappropriate responses
- Contextual grounding: Reduce hallucination by checking against provided context

SKILL 1.6.2: Handling Interactive Conversations
================================================================================
Interactive AI applications—chatbots, assistants, conversational interfaces—require context management across multiple turns. The user's current message only makes sense with previous exchanges.

**DynamoDB** stores conversation history efficiently. Store turns with a session ID key for fast retrieval when constructing prompts. TTL settings automatically expire old conversations, managing storage costs.

**Context window limits** force design decisions. When conversation history exceeds model limits, you must either summarize older turns, drop the oldest, or use retrieval to selectively include relevant history. Each approach trades off context preservation vs. relevance.

**Amazon Comprehend** can extract intent before FM processing. Classify user intent ('complaint,' 'question,' 'feedback') to route to specialized prompts or pre-filter responses. This hybrid approach—traditional NLP for classification, FM for generation—is often more efficient than using FMs for everything.

**Clarification workflows** handle ambiguous input. When the FM detects unclear intent, generate clarifying questions rather than guessing. Step Functions can orchestrate multi-turn clarification flows, tracking state and guiding users toward clear requests.

AWS SERVICES:
- Amazon DynamoDB: Store conversation history with session keys. Fast retrieval for prompt construction. TTL for cleanup.
  Keywords: conversation history, session, context, multi-turn
- Amazon Comprehend: Intent classification before FM processing. Entity extraction from user input.
  Keywords: intent, entity extraction, NLP, classify
- AWS Step Functions: Orchestrate clarification workflows. Multi-turn conversation logic with state management.
  Keywords: workflow, clarification, orchestrate, state machine

SKILL 1.6.3: Implementing Prompt Governance
================================================================================
Prompt governance ensures quality, compliance, and consistency across your organization's AI applications. Without it, ungoverned prompts become liabilities—inconsistent quality, compliance gaps, duplicated effort.

**Parameterized templates** are the foundation. Define approved structures that teams customize through parameters, not by modifying prompt logic. This ensures consistent quality while enabling flexibility. Store templates in Prompt Management with strict access controls.

**S3** can store approved prompt templates as version-controlled files. Combined with S3 versioning and object lock, this provides an immutable audit trail. Organizations with strict compliance requirements may prefer this explicit version control.

**CloudTrail** logs all prompt access and modifications. For audits: Who modified this prompt? When? What was the previous version? Essential for regulated industries where AI decisions must be explainable and traceable.

**Approval workflows** using Step Functions formalize review. When someone submits a prompt change, the workflow routes to reviewers, collects approvals, then promotes to production. Untested prompts can't reach users.

AWS SERVICES:
- Amazon S3: Store approved prompt templates with versioning. Object lock for immutable audit trail.
  Keywords: store, template, version, immutable
- AWS CloudTrail: Audit logging for prompt access and modifications. Compliance tracking.
  Keywords: audit, logging, compliance, track changes
- AWS Step Functions: Approval workflows for prompt changes. Formalized review process.
  Keywords: approval, workflow, review process

SKILL 1.6.4: Ensuring Output Quality
================================================================================
Quality assurance validates that FM outputs meet your standards before delivery. Without QA, you're trusting probabilistic outputs to always be correct, properly formatted, and appropriate—risky assumption.

**Lambda post-processing** runs custom validation. Check prices are positive, dates are valid, referenced products exist in your catalog. Transform outputs into required formats. Enrich responses with additional data. This processing layer catches issues the FM cannot.

**CloudWatch monitoring** tracks quality over time. Log key metrics: format validity rate, business rule pass rate, user satisfaction signals. Set alarms when metrics deviate. This early warning system catches degradation before users complain.

**Step Functions** orchestrates multi-stage validation. Complex validation might require multiple checks: format validation, business rules, safety filtering, manager approval for high-stakes outputs. Step Functions manages this pipeline with proper error handling and retry logic.

**Feedback loops** improve prompts based on output quality. When validation catches systematic issues, that signal should feed back to prompt engineers for improvement. Build mechanisms to collect, aggregate, and route quality signals.

SKILL 1.6.5: Advanced Prompting Techniques
================================================================================
Advanced prompting techniques improve output quality for challenging tasks. These build on fundamentals, addressing specific limitations of standard prompting.

**Chain-of-thought** dramatically improves reasoning accuracy. For math, logic, or multi-step analysis, adding 'Show your reasoning step by step' before asking for the answer can double accuracy rates. The reasoning chain also aids debugging—you can see where logic went wrong.

**Structured inputs** provide consistent context. Rather than varying how you present information, define a consistent format: 'Product: {name}, Price: {price}, Category: {category}.' Predictability helps the model extract information reliably.

**Self-critique** asks the model to review its own output. After generating an initial response, prompt: 'Review your response for accuracy and completeness. Are there any errors or omissions?' The critique often catches issues the initial generation missed.

**Feedback loops** refine prompts based on output quality. Track which prompts produce good outputs and which produce errors. Analyze error patterns to improve wording. This iterative refinement—prompt engineering as continuous practice, not one-time task—is how production prompts achieve high quality.

AWS SERVICES:
- Amazon Bedrock: All FM inference supporting prompting techniques. Experiment with different approaches.
  Keywords: chain-of-thought, structured, advanced prompting, reasoning

**Bedrock Prompt Flows** provides visual orchestration for prompt chains. Connect prompts graphically, define transformations between stages, add conditional branches based on FM outputs. Ideal for rapid development and teams who prefer visual tools.

**Conditional branching** routes to different paths based on intermediate results. A classification stage determines if input is a question, complaint, or feedback, then routes to specialized prompts. Sophisticated, adaptive AI systems.

**Error handling** is essential for production. What happens when the FM returns unexpected format? When a chain fails midway? Build retry logic, fallback paths, and graceful degradation. Step Functions excels at this complex error handling.
