================================================================================
CONCEPT: Bedrock Architecture - How It Actually Works
================================================================================
Amazon Bedrock is a fully managed service that provides API access to foundation models from multiple providers. Understanding its architecture helps you make better design decisions.

**The Bedrock Service Model:**
Bedrock is not hosting models in your account. Instead, it provides a managed API layer that routes your requests to model infrastructure AWS manages. This means:
- No infrastructure to manage (no EC2, no containers)
- Pay-per-use pricing (tokens processed)
- Instant access to multiple model families
- AWS handles scaling, availability, and updates

**Request Flow:**
Your App → Bedrock API → Model Router → Foundation Model → Response
                ↓
         Guardrails (optional)
                ↓
         Logging (optional)

**Key Architecture Components:**
1. Model Access Layer: API endpoints that accept your requests
2. Model Router: Directs requests to the appropriate model
3. Inference Engine: Actually runs the model (managed by AWS)
4. Guardrails Engine: Optional content filtering
5. Logging Service: Optional invocation logging to S3/CloudWatch

**Regional Availability:**
Bedrock is available in specific AWS regions. Not all models are available in all regions. Cross-Region Inference can route requests to other regions for availability and latency optimization.

**VPC Integration:**
By default, Bedrock API calls go over the public internet (with TLS encryption). For private connectivity, create an Interface VPC Endpoint. This keeps traffic within AWS network.

**Authentication:**
All Bedrock API calls require IAM authentication. Use IAM roles for Lambda/EC2, or IAM users with credentials for local development. The principal must have `bedrock:InvokeModel` permission.

KEY POINTS:
- Bedrock is a managed API layer, not model hosting in your account
- Pay-per-token pricing with no infrastructure management
- VPC Endpoints enable private connectivity
- Cross-Region Inference provides availability and latency benefits

EXAM TIP: When questions mention 'managed' or 'no infrastructure' for FM access, think Bedrock over SageMaker.

CONCEPT: Bedrock APIs - The Complete Guide
================================================================================
Understanding the Bedrock APIs is crucial. The exam tests your knowledge of when to use each API and their characteristics.

**InvokeModel API (Synchronous)**

import boto3
import json

client = boto3.client('bedrock-runtime')

response = client.invoke_model(
    modelId='anthropic.claude-3-sonnet-20240229-v1:0',
    contentType='application/json',
    accept='application/json',
    body=json.dumps({
        'anthropic_version': 'bedrock-2023-05-31',
        'max_tokens': 1024,
        'messages': [
            {'role': 'user', 'content': 'Explain quantum computing.'}
        ]
    })
)

result = json.loads(response['body'].read())
print(result['content'][0]['text'])

Key characteristics:
- Synchronous: Request blocks until response is complete
- Best for: Short responses, simple integrations
- Timeout: Default 60 seconds, max varies by model
- Returns: Complete response in single payload

**InvokeModelWithResponseStream API (Streaming)**

response = client.invoke_model_with_response_stream(
    modelId='anthropic.claude-3-sonnet-20240229-v1:0',
    contentType='application/json',
    body=json.dumps({
        'anthropic_version': 'bedrock-2023-05-31',
        'max_tokens': 1024,
        'messages': [
            {'role': 'user', 'content': 'Write a story.'}
        ]
    })
)

for event in response['body']:
    chunk = json.loads(event['chunk']['bytes'])
    if chunk['type'] == 'content_block_delta':
        print(chunk['delta']['text'], end='')

Key characteristics:
- Asynchronous streaming: Tokens arrive as generated
- Best for: Long responses, real-time UIs, chatbots
- Perceived latency: Much better (first token in ~500ms)
- Total time: Same as sync, but user sees progress

**Converse API (Multi-turn)**
Simplified API for conversations:

response = client.converse(
    modelId='anthropic.claude-3-sonnet-20240229-v1:0',
    messages=[
        {'role': 'user', 'content': [{'text': 'Hello!'}]},
        {'role': 'assistant', 'content': [{'text': 'Hi there!'}]},
        {'role': 'user', 'content': [{'text': 'Tell me a joke.'}]}
    ],
    inferenceConfig={'maxTokens': 512, 'temperature': 0.7}
)

Key characteristics:
- Unified format across model families
- Handles conversation history
- Easier than model-specific payload formats

**ApplyGuardrail API**
Apply guardrails without invoking a model:

response = client.apply_guardrail(
    guardrailIdentifier='my-guardrail-id',
    guardrailVersion='1',
    source='INPUT',
    content=[{'text': {'text': 'User input to check'}}]
)

Useful for: Pre-checking user input, post-processing without re-running model.

KEY POINTS:
- InvokeModel: Synchronous, blocks until complete
- InvokeModelWithResponseStream: Streaming for better perceived latency
- Converse API: Simplified multi-turn conversation format
- ApplyGuardrail: Content filtering without model invocation

EXAM TIP: 'Improve perceived latency' or 'real-time chat' → InvokeModelWithResponseStream

CONCEPT: Understanding Model Families & Selection
================================================================================
Bedrock offers models from multiple providers. Each has different strengths, pricing, and use cases.

**Anthropic Claude Models:**
- Claude 3 Opus: Most capable, best for complex reasoning, highest cost
- Claude 3 Sonnet: Balanced capability/cost, good for most production use
- Claude 3 Haiku: Fastest, cheapest, best for simple tasks
- Claude 3.5 Sonnet: Latest, improved over Claude 3 Sonnet

Best for: Complex reasoning, analysis, code generation, long-form content
Context window: Up to 200K tokens

**Amazon Titan Models:**
- Titan Text: General-purpose text generation
- Titan Embeddings: Text-to-vector for semantic search (critical for RAG)
- Titan Multimodal Embeddings: Image + text embeddings

Best for: Cost-effective tasks, embeddings for RAG
Note: Titan models are optimized for AWS infrastructure

**Meta Llama Models:**
- Llama 3: Open-source, good general performance
- Llama 3.1: Larger context, improved capabilities

Best for: When open-source alignment matters, cost-sensitive use cases

**Mistral Models:**
- Mistral 7B: Compact, efficient
- Mixtral 8x7B: Mixture of experts, good cost/performance

Best for: Efficient inference, specific use cases

**Cohere Models:**
- Command: Text generation
- Embed: Embeddings for search
- Rerank: Reranking search results

Best for: Enterprise search, RAG reranking

**Model Selection Decision Tree:**
Need embeddings?
  → Yes: Titan Embeddings or Cohere Embed
  → No: Continue...

Need complex reasoning?
  → Yes: Claude 3 Opus or Claude 3.5 Sonnet
  → No: Continue...

Need fastest/cheapest?
  → Yes: Claude 3 Haiku or Mistral 7B
  → No: Claude 3 Sonnet (balanced default)

**Model IDs (Critical for API calls):**
- anthropic.claude-3-sonnet-20240229-v1:0
- anthropic.claude-3-haiku-20240307-v1:0
- amazon.titan-text-express-v1
- amazon.titan-embed-text-v1

**Inference Profiles:**
Bedrock allows creating inference profiles that abstract model selection. Your code calls the profile; you can switch models without code changes.

KEY POINTS:
- Claude: Best for reasoning and complex tasks (Opus > Sonnet > Haiku)
- Titan Embeddings: Essential for RAG, optimized for AWS
- Model IDs are required for API calls
- Inference profiles enable model switching without code changes

EXAM TIP: Questions often present scenarios and ask which model. Match: complex reasoning → Claude, embeddings → Titan, cost-sensitive simple → Haiku

CONCEPT: Knowledge Bases - Managed RAG Infrastructure
================================================================================
Knowledge Bases provide managed RAG (Retrieval-Augmented Generation) without building your own pipeline.

**What Knowledge Bases Handle:**
1. Document Ingestion: Read documents from S3
2. Chunking: Split documents into processable pieces
3. Embedding Generation: Convert chunks to vectors
4. Vector Storage: Store in managed or your own vector DB
5. Retrieval: Find relevant chunks for queries
6. Generation: Send context to FM for response

**Data Sources:**
- Amazon S3 (documents, PDFs, text files)
- Web crawlers
- Confluence
- SharePoint
- Salesforce
- Custom via API

**Vector Store Options:**
Managed Vector Store (Simplest)
  → AWS manages everything
  → Good for: Getting started, small/medium scale

Amazon OpenSearch Serverless
  → Serverless, scalable
  → Good for: Production, hybrid search needs

Amazon Aurora (pgvector)
  → Relational + vector
  → Good for: Existing Aurora users, SQL queries

Pinecone / Redis Enterprise
  → Third-party options
  → Good for: Specific requirements

**Chunking Strategies:**
- Fixed-size chunking: chunk_size = 300 tokens, overlap = 50 tokens
- Semantic chunking: Splits at natural boundaries (paragraphs, sections)
- Hierarchical chunking: Parent-child relationships preserved

**APIs for Knowledge Bases:**

RetrieveAndGenerate (Full RAG in one call):
response = bedrock_agent.retrieve_and_generate(
    input={'text': 'What is our refund policy?'},
    retrieveAndGenerateConfiguration={
        'type': 'KNOWLEDGE_BASE',
        'knowledgeBaseConfiguration': {
            'knowledgeBaseId': 'KB_ID',
            'modelArn': 'arn:aws:bedrock:us-east-1::foundation-model/anthropic.claude-3-sonnet'
        }
    }
)

Retrieve (Retrieval only - you handle generation):
response = bedrock_agent.retrieve(
    knowledgeBaseId='KB_ID',
    retrievalQuery={'text': 'refund policy'},
    retrievalConfiguration={
        'vectorSearchConfiguration': {
            'numberOfResults': 5
        }
    }
)

When to use each:
- RetrieveAndGenerate: Simple RAG, let AWS handle everything
- Retrieve: Need custom prompts, multi-step processing, or non-Bedrock FMs

KEY POINTS:
- Knowledge Bases handle the full RAG pipeline: ingest, chunk, embed, store, retrieve, generate
- Choose vector store based on scale and requirements
- RetrieveAndGenerate for simple RAG, Retrieve for custom processing
- Chunking strategy affects retrieval quality

EXAM TIP: 'Managed RAG' or 'RAG without building pipeline' → Knowledge Bases

CONCEPT: Agents - Autonomous AI Systems
================================================================================
Bedrock Agents enable autonomous AI that can reason, plan, and take actions using external tools.

**How Agents Work:**
User Query → Agent FM (Claude) → Reasoning
                    ↓
            Plan: Which actions needed?
                    ↓
            Execute: Call Action Groups
                    ↓
            Observe: Process results
                    ↓
            Iterate or Respond

This is the **ReAct pattern**: Reason → Act → Observe → Repeat

**Agent Components:**
1. Instructions: System prompt that defines agent behavior
2. Action Groups: Tools the agent can use
3. Knowledge Bases: Documents agent can query
4. Guardrails: Safety controls (optional)

**Action Groups:**
Each action is backed by a Lambda function:

OpenAPI Schema for Action:
openapi: 3.0.0
paths:
  /getWeather:
    get:
      summary: Get current weather
      operationId: getWeather
      parameters:
        - name: city
          in: query
          required: true
          schema:
            type: string

Lambda function:
def handler(event, context):
    action = event['actionGroup']
    api_path = event['apiPath']
    parameters = event['parameters']

    city = next(p['value'] for p in parameters if p['name'] == 'city')
    weather_data = get_weather(city)

    return {
        'actionGroup': action,
        'apiPath': api_path,
        'httpMethod': 'GET',
        'httpStatusCode': 200,
        'responseBody': {
            'application/json': {
                'body': json.dumps(weather_data)
            }
        }
    }

**Invoking Agents:**
response = bedrock_agent_runtime.invoke_agent(
    agentId='AGENT_ID',
    agentAliasId='ALIAS_ID',
    sessionId='unique-session-id',
    inputText='What\'s the weather in Seattle?'
)

for event in response['completion']:
    if 'chunk' in event:
        print(event['chunk']['bytes'].decode(), end='')

**Agent Tracing:**
Enable tracing to see the agent's reasoning:
- Which tools it considered
- Which tools it called
- What results it received
- How it formulated the response

Essential for debugging and understanding agent behavior.

**Multi-Agent Collaboration (Agent Squad):**
Orchestrate multiple specialized agents:
- Supervisor agent routes to specialists
- Each agent has specific expertise
- Results aggregated for final response

KEY POINTS:
- Agents use ReAct pattern: Reason → Act → Observe → Repeat
- Action Groups are Lambda functions with OpenAPI schemas
- Session IDs maintain conversation context
- Tracing reveals agent reasoning for debugging

EXAM TIP: 'Autonomous' or 'tool calling' or 'multi-step reasoning' → Bedrock Agents

CONCEPT: Guardrails - Safety and Control
================================================================================
Guardrails provide content filtering, topic control, and PII protection for Bedrock applications.

**Guardrail Components:**

1. Content Filters: Block harmful content categories
   - Hate speech
   - Insults
   - Sexual content
   - Violence
   - Misconduct
   Each category has configurable strength (NONE, LOW, MEDIUM, HIGH)

2. Denied Topics: Block specific conversation topics
   Topic: Medical Diagnosis
   Sample phrases:
   - "What disease do I have?"
   - "Should I take this medication?"
   - "Diagnose my symptoms"

3. Word Filters: Block specific words/phrases
   - Profanity
   - Competitor names
   - Sensitive terms

4. PII Detection: Identify and handle personal information
   - Names, addresses, phone numbers
   - SSN, credit cards, bank accounts
   - Email addresses, IP addresses
   Actions: BLOCK or ANONYMIZE

**Applying Guardrails:**

With model invocation:
response = client.invoke_model(
    modelId='anthropic.claude-3-sonnet-20240229-v1:0',
    guardrailIdentifier='my-guardrail-id',
    guardrailVersion='DRAFT',
    body=json.dumps({...})
)

Standalone (ApplyGuardrail API):
response = client.apply_guardrail(
    guardrailIdentifier='my-guardrail-id',
    guardrailVersion='1',
    source='INPUT',  # or 'OUTPUT'
    content=[{'text': {'text': 'Check this content'}}]
)

if response['action'] == 'GUARDRAIL_INTERVENED':
    print('Content blocked:', response['outputs'])

**Guardrail Responses:**
- NONE: Content passed all checks
- GUARDRAIL_INTERVENED: Content was modified or blocked

**Use Cases:**
- Customer-facing chatbots (filter harmful content)
- Enterprise applications (prevent data leakage)
- Compliance (block regulated topics)
- Brand protection (filter competitor mentions)

**Defense in Depth:**
Guardrails are one layer. Combine with:
- Input validation (Lambda pre-processing)
- Output validation (Lambda post-processing)
- Prompt design (clear boundaries in system prompt)
- Monitoring (CloudWatch alerts on violations)

KEY POINTS:
- Content Filters block harmful categories with configurable strength
- Denied Topics prevent specific conversation subjects
- PII Detection can block or anonymize personal data
- Apply to both INPUT and OUTPUT for complete protection

EXAM TIP: 'Content filtering', 'PII protection', or 'topic blocking' → Bedrock Guardrails

CONCEPT: Pricing Models and Cost Optimization
================================================================================
Understanding Bedrock pricing is essential for cost optimization and the exam.

**On-Demand Pricing:**
Pay per token processed. No commitment required.

Claude 3 Haiku:    $0.25 / 1M input tokens,  $1.25 / 1M output tokens
Claude 3 Sonnet:   $3.00 / 1M input tokens,  $15.00 / 1M output tokens
Claude 3 Opus:     $15.00 / 1M input tokens, $75.00 / 1M output tokens
Titan Text:        $0.20 / 1M input tokens,  $0.60 / 1M output tokens
Titan Embeddings:  $0.10 / 1M tokens

Note: Output tokens cost more because they require sequential generation.

**Provisioned Throughput:**
Commit to capacity for lower per-token costs.
- Purchase Model Units (capacity)
- 1-month or 6-month commitments
- Lower effective per-token cost at high utilization
- Break-even typically around 40-60% utilization

When to use:
- Predictable, sustained workloads
- Need guaranteed latency/throughput
- High volume (cost savings outweigh commitment)

**Batch Inference:**
Process large volumes asynchronously at reduced cost.
- Submit batch jobs (files of requests)
- Results delivered to S3
- ~50% cost reduction vs on-demand
- Not real-time (hours turnaround)

When to use:
- Bulk processing
- Offline analysis
- Non-time-sensitive workloads

**Cost Optimization Strategies:**

1. Model Selection: Use cheapest model that meets quality needs
   Simple FAQ → Haiku ($0.25/1M)
   Complex analysis → Sonnet ($3/1M)
   Critical reasoning → Opus ($15/1M)

2. Token Optimization:
   - Shorter prompts (remove redundancy)
   - Limit output length (max_tokens)
   - Summarize context instead of full history

3. Caching:
   - Prompt caching (reuse processed prefixes)
   - Semantic caching (match similar queries)
   - Edge caching (CloudFront for repeated responses)

4. Model Cascading:
   Query → Classifier (cheap)
             ↓
   Simple → Haiku | Complex → Sonnet

KEY POINTS:
- Output tokens cost more than input tokens
- Provisioned Throughput saves money at 40%+ utilization
- Batch inference for non-real-time bulk processing
- Model cascading routes queries to appropriate cost tier

EXAM TIP: 'Cost optimization' → model selection, caching, provisioned throughput for sustained loads

SERVICE COMPARISON: Bedrock vs SageMaker for FM Deployment
================================================================================
| Criterion      | Amazon Bedrock                              | Amazon SageMaker                           |
|----------------|---------------------------------------------|-------------------------------------------|
| Infrastructure | Fully managed, no infrastructure            | Managed endpoints, you configure instances |
| Model Selection| Pre-selected foundation models from providers| Any model you can containerize            |
| Customization  | Fine-tuning via Bedrock                     | Full control over training and deployment  |
| Best For       | Using foundation models with minimal ops    | Custom models, full infrastructure control |

================================================================================
EXAM STRATEGIES
================================================================================
Pattern: "managed FM access without infrastructure"
Approach: Amazon Bedrock. It's the managed service for foundation models without infrastructure management.
Keywords: managed, no infrastructure, foundation model, API access

Pattern: "improve response latency for chatbot"
Approach: InvokeModelWithResponseStream for streaming. First token arrives quickly even if full response takes time.
Keywords: latency, chatbot, streaming, real-time

Pattern: "content filtering or safety"
Approach: Bedrock Guardrails. Configure content filters, denied topics, and PII detection.
Keywords: content filter, safety, PII, guardrails, block content

================================================================================
KEY TAKEAWAYS
================================================================================
1. Bedrock is a managed API layer - no infrastructure in your account
2. InvokeModel for sync, InvokeModelWithResponseStream for streaming
3. Knowledge Bases provide managed RAG with chunking, embedding, and retrieval
4. Agents enable autonomous tool calling with ReAct pattern
5. Guardrails filter content, block topics, and protect PII
6. Model selection: Haiku (cheap/fast) → Sonnet (balanced) → Opus (best reasoning)

================================================================================
COMMON MISTAKES
================================================================================
- Using InvokeModel instead of streaming for long responses (poor UX)
- Not enabling Guardrails for user-facing applications
- Using expensive models (Opus) for simple tasks that Haiku handles
- Forgetting VPC Endpoints when private connectivity is required
- Not considering Provisioned Throughput for sustained high-volume workloads
