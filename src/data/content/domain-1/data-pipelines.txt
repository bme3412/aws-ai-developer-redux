DATA PIPELINES FOR GENAI
Domain 1 | Task 1.3 | Estimated: 30 minutes

================================================================================
WHY IT MATTERS
================================================================================
Garbage in, garbage out—this old rule applies double for AI. Feed your model messy data, and you'll get messy responses. But clean, well-formatted inputs? That's how you get impressive results without changing a single line of your prompt. This topic is about everything that happens to data before it reaches the model.

================================================================================
CONCEPT: Data Quality for GenAI
================================================================================
Here's the frustrating thing about AI: it won't tell you when input data is bad. It'll just produce garbage output and act like everything's fine. That's why validation matters so much.

**Format validation** catches the obvious stuff. Is the text actually UTF-8? Is that JSON parseable? Is the image actually an image, not a corrupted file? These sound basic, but you'd be amazed what shows up in production data.

**Completeness checking** ensures nothing's missing. Your prompt template expects a customer name and order history? Make sure both actually exist before building the prompt. Missing data leads to weird, generic responses—or worse, the model invents information to fill the gap.

**Anomaly detection** flags the weird stuff. A product price of -$50? A date in year 2099? An email that's 500 characters long? Something's wrong. Domain-specific rules catch these issues.

**AWS Glue Data Quality** handles batch validation with declarative rules. Write rules like 'price > 0' or 'email contains @' in DQDL syntax. Run them automatically in your Glue ETL jobs. If data fails quality checks, stop the pipeline before wasting money on bad AI calls.

**SageMaker Data Wrangler** is your visual Swiss Army knife for data prep. Explore distributions, spot anomalies, build transformation recipes—all through a GUI. Great for understanding your data before building production pipelines.

For **real-time validation**, Lambda functions are your friend. Custom business logic that checks inputs the moment they arrive. Reject bad data immediately instead of discovering problems after expensive processing.

KEY POINTS:
- Format: Valid encoding, parseable JSON, correct file types
- Completeness: All required fields present, no truncation
- Anomalies: Catch impossible values and obvious errors
- Glue Data Quality: Batch rules; Lambda: Real-time custom checks

EXAM TIP: 'Data quality rules' or 'batch validation' = Glue Data Quality. 'Real-time custom validation' = Lambda.

================================================================================
CONCEPT: Multimodal Data Processing
================================================================================
Modern AI doesn't just read text—it sees images, reads documents, and hears audio. But each type needs specific prep work before the model can use it.

**Images** need resizing and encoding. Models have resolution limits, so scale down massive images. Convert to supported formats (JPEG, PNG, GIF, WebP). Encode as **base64** for API transmission. For Bedrock's multimodal models (Claude 3, Titan), images go directly into the message content alongside text.

**Documents** (PDFs, Word files, scanned pages) need text extraction. **Amazon Textract** is your OCR powerhouse with multiple APIs:
- DetectDocumentText: Basic OCR, returns lines and words
- AnalyzeDocument: Extracts forms (key-value pairs) and tables with cell relationships
- AnalyzeExpense: Specialized for receipts and invoices
- AnalyzeID: Extracts data from identity documents

Textract preserves document structure that simple OCR loses—critical for maintaining context in RAG systems.

**Audio** becomes text through **Amazon Transcribe**. Key features:
- Speaker diarization: Identifies different speakers (who said what)
- Custom vocabularies: Improve accuracy for domain jargon
- Real-time streaming: For live transcription
- Batch processing: For recorded audio at scale

**Video** is the combo meal—extract key frames for image analysis, transcribe the audio track for spoken content. Some workflows analyze both and synthesize insights.

**Bedrock Data Automation** (newer service) can automate document processing pipelines—worth knowing exists.

Critical: **handle errors gracefully**. Corrupted images, garbled audio, password-protected PDFs—they all happen. Skip bad inputs, log issues, continue processing. Don't let one bad file crash your entire pipeline.

KEY POINTS:
- Textract APIs: DetectDocumentText, AnalyzeDocument (forms/tables), AnalyzeExpense, AnalyzeID
- Transcribe: Speaker diarization, custom vocabularies, real-time or batch
- Images: Base64 encode, use JPEG/PNG/GIF/WebP formats
- Error handling: Skip bad inputs gracefully, don't crash pipelines

EXAM TIP: 'Forms or tables' → Textract AnalyzeDocument. 'Speaker identification' → Transcribe with diarization. 'Receipts' → Textract AnalyzeExpense.

================================================================================
CONCEPT: API-Ready Data Formatting
================================================================================
AI APIs are picky about input format. Get it wrong and your request fails—sometimes with cryptic errors that take forever to debug.

**Bedrock's Messages API** uses a specific structure: messages with roles ('user' or 'assistant'), content arrays that can include text and images, and a system message for instructions. It looks like:

{messages: [{role: 'user', content: [{type: 'text', text: '...'}]}]}

Miss a bracket? Wrong field name? Improper nesting? Request fails.

**Token limits** are real constraints. Every model has a maximum context size. If your input exceeds it, the request fails or gets truncated. Know your model's limits. Options for oversized inputs: truncate (lose data), summarize (adds processing), or chunk and aggregate (complex but complete).

**Special characters** need proper escaping in JSON. Newlines, quotes, backslashes, control characters—they'll all break your JSON if not escaped correctly. Use proper serialization libraries, not string concatenation.

**SageMaker endpoints** have custom schemas. Each endpoint defines its own expected format. Document these clearly and validate inputs against them.

Best practice: **build a formatting layer**. Don't construct JSON throughout your codebase. Create utility functions that handle serialization consistently. When the API changes (and it will), you fix one place.

KEY POINTS:
- Messages API: Roles + content arrays + system message
- Token limits: Know your model's limits, handle oversized inputs
- Special characters: Use proper JSON serialization, not string concat
- Consistency: Build a formatting layer, don't repeat yourself

================================================================================
CONCEPT: Input Enhancement Techniques
================================================================================
Raw user input is often... raw. Typos, abbreviations, missing context. You can improve AI responses dramatically by preprocessing input before it reaches the model.

**Entity extraction** identifies the important stuff. **Amazon Comprehend** pulls out names, dates, locations, and organizations. It detects sentiment and key phrases. Use this info to route requests, filter responses, or add context to prompts.

**Text normalization** cleans things up. Expand 'u' to 'you'. Fix obvious typos. Standardize date formats. Remove excessive punctuation. The model handles imperfect input, but cleaner input often produces cleaner output.

**Context enrichment** adds background. User asks about 'my order'? Inject their recent order details into the prompt. References a product by nickname? Resolve it to the full name. This context enables specific, helpful responses instead of generic ones.

**Query expansion** broadens retrieval. A question about 'compute options' might expand to include 'EC2', 'Lambda', 'Fargate'. This helps RAG systems find relevant documents that use different terminology.

**Personalization** tailors responses. Include relevant user history, preferences, or role. A developer might want technical details; an exec might want a summary.

These enhancements run **before** the AI call, typically in Lambda. They transform raw input into optimized prompts. But measure effectiveness—not all preprocessing helps, and some adds latency without improving results.

KEY POINTS:
- Entity extraction: Comprehend pulls names, dates, sentiment
- Normalization: Fix typos, expand abbreviations, standardize
- Context injection: Add user history, resolve references
- Query expansion: Add synonyms for better retrieval

SKILL 1.3.1: Validating Input Data
================================================================================
Validation catches problems before they become expensive mistakes.

**Glue Data Quality** defines rules declaratively. Write rules like 'Completeness of customer_id > 99%' or 'price between 0 and 10000'. Rules run automatically in Glue ETL jobs. If data fails, the pipeline stops—you don't waste money processing garbage.

**Data Wrangler** helps you understand data visually. View distributions, spot outliers, explore relationships. It's great for building intuition about your data before writing validation rules. Export recipes to production pipelines.

**Lambda** handles real-time custom validation. Check that referenced products exist in your catalog. Verify prices fall within valid ranges. Confirm customer IDs match authenticated users. Business-specific rules that generic tools can't express.

Design validation to **fail fast**. If input is bad, reject immediately. Return clear error messages so users can fix their input. Don't let bad data travel through expensive processing only to produce garbage at the end.

AWS SERVICES:
- AWS Glue Data Quality: Batch validation with declarative rules. Quality monitoring for data lakes.
  Keywords: data quality, validation rules, completeness, DQDL
- Amazon SageMaker Data Wrangler: Visual exploration. Understanding data before building pipelines.
  Keywords: data wrangler, visual, transformation, explore
- AWS Lambda: Real-time custom validation. Business rules per request.
  Keywords: custom, real-time, business rules

SKILL 1.3.2: Preparing Multimodal Inputs
================================================================================
Each input type needs specific preparation to work with AI models.

**Bedrock multimodal models** accept images directly in the message content. Base64-encode them, keep them within size limits, and they're ready. The model can analyze visual content, answer questions about images, or describe what it sees.

**SageMaker Processing** handles heavy batch transformations. Need to preprocess 10,000 images? Processing jobs run on managed infrastructure that scales automatically. More efficient than Lambda for high-volume batch work.

**Textract** extracts text while preserving structure. For forms and tables, it identifies fields and their values. The structured output feeds directly into AI prompts for document analysis.

**Transcribe** converts speech to text with speaker identification. Custom vocabularies improve accuracy for your industry's jargon. Real-time mode works for live apps; batch handles recordings.

Remember: each modality has failure modes. Corrupted images, garbled audio, locked PDFs. Build error handling into your pipeline.

AWS SERVICES:
- Amazon Bedrock: Multimodal FMs accepting images directly. Vision analysis.
  Keywords: multimodal, image understanding, vision
- Amazon Textract: Document OCR. Form and table extraction.
  Keywords: textract, OCR, document, forms
- Amazon Transcribe: Speech to text. Audio processing.
  Keywords: transcribe, audio, speech-to-text

================================================================================
SKILL 1.3.3: Formatting Data for FM APIs
================================================================================
Getting the format right means your requests work. Getting it wrong means cryptic errors.

**Bedrock's Messages API** is your interface to most foundation models. Learn its structure: system messages for instructions, user/assistant roles for conversation, content arrays for multimodal inputs. Build utilities that construct valid messages reliably.

**Token counting** before sending requests prevents surprises. If input exceeds limits, decide how to handle it: truncate (simple but loses data), summarize first (adds processing), or chunk with aggregation (complex but complete).

**Error handling** catches the inevitable failures. Parse errors, encoding issues, malformed responses—they all happen. Build robust handling that logs problems, returns meaningful errors, and makes debugging possible.

The key insight: **centralize formatting logic**. Don't build JSON in ten different places. Create a formatting layer that everyone uses. When APIs change, you update one file.

AWS SERVICES:
- Amazon Bedrock: Messages API for FM invocation. Standard interface.
  Keywords: messages API, invoke model, converse

================================================================================
SKILL 1.3.4: Enhancing Input Quality
================================================================================
Enhancement makes raw input work better without changing the model.

**Comprehend** extracts entities and sentiment from text. Use it for routing (is this a complaint?), filtering (is sentiment negative?), or context (what product are they discussing?). It's faster and cheaper than using an FM for these standard NLP tasks.

**Lambda** implements custom enhancement. Normalize text, expand abbreviations, inject user context, enrich with database lookups. Business-specific processing that generic tools can't handle.

**Context injection** is powerful. A generic 'what's my order status?' becomes specific when you inject: 'Customer John Smith (ID: 12345) has order #98765, shipped yesterday, arriving Friday.' The AI can give a useful, personalized response.

**Measure what works**. Not all preprocessing improves results. Some adds latency without benefit. A/B test enhanced vs. raw inputs. Focus effort where it demonstrably helps.

AWS SERVICES:
- Amazon Comprehend: Entity extraction, sentiment, key phrases. Standard NLP tasks.
  Keywords: entity extraction, NLP, sentiment, comprehend
- AWS Lambda: Custom text processing. Context injection. Enrichment.
  Keywords: preprocess, normalize, enhance

KEY TAKEAWAYS
================================================================================
1. Validate BEFORE sending to AI—catch errors early, save money
2. Glue Data Quality = batch rules; Lambda = real-time custom logic
3. Comprehend for entities/sentiment—cheaper than FM calls
4. Textract for documents, Transcribe for audio, Bedrock for images
5. Build a formatting layer—don't repeat JSON construction everywhere

================================================================================
COMMON MISTAKES
================================================================================
- Skipping validation (AI produces garbage, not errors)
- Using expensive FMs for tasks Comprehend handles better
- Forgetting token limits (inputs get truncated or fail)
- Not base64-encoding images correctly for multimodal
- Building formatting logic everywhere instead of centralizing
