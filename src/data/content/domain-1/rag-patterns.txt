================================================================================
WHY IT MATTERS
================================================================================
RAG (Retrieval-Augmented Generation) is how you make foundation models knowledgeable about your data. This deep dive covers RAG architecture patterns, chunking strategies, retrieval optimization, and production best practices.

================================================================================
CONCEPT: RAG Fundamentals - Why and How
================================================================================
RAG solves a fundamental problem: foundation models have knowledge cutoffs and don't know about your private data.

**The Problem RAG Solves:**
- FMs trained on public data up to a cutoff date
- FMs don't know your company's documents, policies, products
- FMs can hallucinate when they don't know something
- Fine-tuning is expensive and doesn't update easily

**How RAG Works:**

1. INDEXING (Offline)
   Documents → Chunking → Embedding → Vector Store

2. RETRIEVAL (Runtime)
   User Query → Embedding → Vector Search → Relevant Chunks

3. GENERATION (Runtime)
   Relevant Chunks + Query → FM → Grounded Response

**Why RAG Over Fine-Tuning:**

| Aspect           | RAG                              | Fine-Tuning                    |
|------------------|----------------------------------|--------------------------------|
| Knowledge updates| Re-index documents               | Re-train model                 |
| Cost             | Vector storage + retrieval       | Training compute               |
| Latency          | Retrieval adds ~100-500ms        | No additional latency          |
| Accuracy         | Cites sources, verifiable        | Embedded in weights            |
| Best for         | Factual Q&A, documents           | Style/format changes           |

**RAG Quality Factors:**
1. Retrieval Quality: Are you finding the right chunks?
2. Context Quality: Does the retrieved context contain the answer?
3. Generation Quality: Does the FM use the context correctly?

Each step can fail independently. Debugging requires checking each step.

KEY POINTS:
- RAG grounds FM responses in your documents
- Three steps: Index (offline), Retrieve (runtime), Generate (runtime)
- RAG is cheaper and more updatable than fine-tuning for factual knowledge
- Quality depends on retrieval, context, and generation

EXAM TIP: 'Ground responses in documents' or 'reduce hallucination with company data' → RAG

CONCEPT: Chunking Strategies - The Foundation of Good RAG
================================================================================
Chunking determines how documents are split for embedding and retrieval. Bad chunking = bad RAG.

**Why Chunking Matters:**
- Embeddings have fixed dimensions (capture limited meaning)
- Too large: chunks become semantic mush
- Too small: lose context, can't answer questions
- Split wrong: information gets separated from its context

**Fixed-Size Chunking:**

def fixed_size_chunk(text, chunk_size=500, overlap=50):
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunks.append(text[start:end])
        start = end - overlap  # Overlap preserves context
    return chunks

Pros: Simple, predictable
Cons: Splits mid-sentence, ignores document structure
Best for: Uniform text without clear structure

**Semantic Chunking:**
Split at natural boundaries: Paragraphs, Sections (headers), Sentences (for short content)

def semantic_chunk(text):
    # Split on double newlines (paragraphs)
    paragraphs = text.split('\n\n')

    chunks = []
    current_chunk = ''

    for para in paragraphs:
        if len(current_chunk) + len(para) < 1000:
            current_chunk += para + '\n\n'
        else:
            chunks.append(current_chunk.strip())
            current_chunk = para + '\n\n'

    if current_chunk:
        chunks.append(current_chunk.strip())

    return chunks

Pros: Preserves meaning, natural boundaries
Cons: Variable chunk sizes, may need post-processing
Best for: Structured documents (articles, reports)

**Hierarchical Chunking:**
Create parent-child relationships:

Document
  └── Section (parent chunk)
        └── Paragraph 1 (child chunk)
        └── Paragraph 2 (child chunk)

Retrieval: Find relevant child, include parent for context.

Pros: Maintains document structure, good context
Cons: More complex indexing and retrieval
Best for: Technical documentation, legal documents

**Bedrock Knowledge Base Chunking Options:**
- Default: Fixed-size with semantic awareness
- Custom: Define your own chunk size and overlap
- Hierarchical: Parent-child relationships (advanced)

**Chunk Size Guidelines:**
100-200 tokens: Very granular, good for specific facts
300-500 tokens: Balanced, good default for most use cases
500-1000 tokens: More context, better for complex topics
1000+ tokens: Risk of diluted embedding, poor precision

**Overlap Strategy:**
Overlap prevents losing context at boundaries:
Chunk 1: "...the quarterly revenue was $50M. The main driver"
Chunk 2: "$50M. The main driver was the new product line..."

Typical overlap: 10-20% of chunk size.

KEY POINTS:
- Chunk size affects retrieval precision (smaller) vs context (larger)
- Semantic chunking preserves natural document boundaries
- Hierarchical chunking maintains document structure
- Overlap prevents losing context at chunk boundaries

EXAM TIP: 'Improve retrieval accuracy' or 'chunking strategy' → match chunking to document structure, consider overlap

CONCEPT: Embeddings - Converting Text to Vectors
================================================================================
Embeddings convert text into numerical vectors that capture semantic meaning. Similar meanings = similar vectors.

**How Embeddings Work:**
"The cat sat on the mat"  →  [0.23, -0.45, 0.12, ..., 0.89]  (1536 dimensions)
"A feline rested on a rug" →  [0.21, -0.43, 0.14, ..., 0.87]  (similar vector!)
"Stock prices rose sharply" →  [0.78, 0.34, -0.56, ..., 0.12]  (different vector)

**Vector Similarity:**
- Cosine similarity: Angle between vectors (most common)
- Dot product: Magnitude matters
- Euclidean distance: Straight-line distance

Cosine similarity of 0.95+ = very similar meaning.

**Amazon Titan Embeddings:**

import boto3
import json

client = boto3.client('bedrock-runtime')

response = client.invoke_model(
    modelId='amazon.titan-embed-text-v1',
    body=json.dumps({
        'inputText': 'What is the refund policy?'
    })
)

result = json.loads(response['body'].read())
embedding = result['embedding']  # 1536-dimensional vector

**Titan Embeddings v1 vs v2:**
- v1: 1536 dimensions, good general performance
- v2: Configurable dimensions (256-1024), normalized

**Batch Embedding Generation:**
For indexing large document collections:

def batch_embed(texts, batch_size=25):
    embeddings = []
    for i in range(0, len(texts), batch_size):
        batch = texts[i:i+batch_size]
        # Process batch...
    return embeddings

**Embedding Model Selection:**

| Model              | Dimensions | Best For                        |
|--------------------|------------|----------------------------------|
| Titan Embeddings v1| 1536       | General purpose, AWS-optimized   |
| Titan Embeddings v2| 256-1024   | Customizable, cost-optimized     |
| Cohere Embed       | 1024       | Multi-lingual                    |

**Critical Rule:**
WARNING: Query and document embeddings MUST use the same model!

If you index with Titan v1, you must query with Titan v1. Different models produce incompatible vector spaces.

**Embedding Quality Factors:**
- Model choice (different models capture different nuances)
- Text preprocessing (cleaning helps quality)
- Chunk quality (garbage in = garbage out)

KEY POINTS:
- Embeddings convert text to vectors where similar meanings are close
- Titan Embeddings is the primary choice for Bedrock RAG
- Query and document embeddings MUST use the same model
- Cosine similarity measures vector closeness

EXAM TIP: 'Semantic search' or 'find similar meaning' → embeddings + vector similarity

CONCEPT: Vector Stores - Where Embeddings Live
================================================================================
Vector stores are databases optimized for similarity search over embeddings.

**How Vector Search Works:**
1. Store document embeddings with metadata
2. Query: Convert query to embedding
3. Search: Find k nearest vectors (most similar)
4. Return: Documents associated with those vectors

**AWS Vector Store Options:**

**Amazon OpenSearch Service:**

# Index document with embedding
index_body = {
    'text': 'Original document text...',
    'embedding': [0.23, -0.45, ...],  # 1536 dims
    'metadata': {
        'source': 'policy.pdf',
        'page': 5
    }
}

os_client.index(index='documents', body=index_body)

# Search for similar
search_body = {
    'size': 5,
    'query': {
        'knn': {
            'embedding': {
                'vector': query_embedding,
                'k': 5
            }
        }
    }
}

Pros: Scalable, hybrid search (keyword + vector), mature
Cons: More configuration required
Best for: Production RAG, complex search requirements

**OpenSearch Serverless:**
Serverless version of OpenSearch:
- Auto-scaling
- Pay per use
- Simpler operations

**Bedrock Knowledge Bases Managed Store:**
- Fully managed by AWS
- No configuration needed
- Good for getting started

**Amazon Aurora (pgvector):**

CREATE EXTENSION vector;

CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    content TEXT,
    embedding vector(1536)
);

-- Search
SELECT content, embedding <=> query_embedding AS distance
FROM documents
ORDER BY embedding <=> query_embedding
LIMIT 5;

Pros: SQL interface, existing Aurora users
Cons: Less optimized for pure vector search
Best for: Teams already using Aurora, hybrid data needs

**Choosing a Vector Store:**

Getting started / Simple RAG
  → Bedrock Knowledge Bases managed store

Production / Scale / Hybrid search
  → OpenSearch Service or Serverless

Existing relational data / SQL preference
  → Aurora pgvector

**Vector Index Types:**
- Flat: Exact search, slow for large datasets
- HNSW: Approximate, fast, good recall (most common)
- IVF: Approximate, cluster-based

OpenSearch uses HNSW by default - good balance of speed and accuracy.

KEY POINTS:
- OpenSearch is primary choice for production vector search
- Knowledge Bases managed store is simplest to start
- Aurora pgvector integrates vectors with relational data
- HNSW index provides fast approximate nearest neighbor search

EXAM TIP: 'Vector database' or 'semantic search storage' → OpenSearch Service for production, Knowledge Bases managed for simple

CONCEPT: Retrieval Optimization - Finding the Right Content
================================================================================
Retrieval quality determines RAG quality. Optimize how you find relevant documents.

**Retrieval Metrics:**
- Precision: % of retrieved docs that are relevant
- Recall: % of relevant docs that were retrieved
- MRR (Mean Reciprocal Rank): How high do relevant docs rank?

**Improving Precision (fewer irrelevant results):**

1. Better chunking: Right-size chunks for your content
2. Similarity threshold: Filter low-similarity results
   results = [r for r in results if r.score > 0.7]
3. Metadata filtering: Pre-filter by known attributes
   results = search(
       query_embedding,
       filter={'source': 'policy_docs', 'year': 2024}
   )

**Improving Recall (not missing relevant docs):**

1. Query expansion: Reformulate query multiple ways
   queries = [
       original_query,
       rephrase_with_llm(original_query),
       expand_acronyms(original_query)
   ]
   results = union([search(q) for q in queries])

2. Hybrid search: Combine vector + keyword
   {
     "query": {
       "bool": {
         "should": [
           {"match": {"text": "refund policy"}},
           {"knn": {"embedding": {"vector": [...], "k": 10}}}
         ]
       }
     }
   }

3. Retrieve more, rerank: Get top 20, rerank to top 5

**Reranking with Bedrock:**
Reranking re-scores results using a more sophisticated model:

# Initial retrieval: fast, approximate
initial_results = vector_search(query, k=20)

# Rerank: slower, more accurate
reranked = bedrock.rerank(
    modelId='cohere.rerank-v3',
    query=query,
    documents=[r.text for r in initial_results],
    topN=5
)

When to use: When precision matters more than latency (e.g., critical decisions).

**Hybrid Search Weighting:**
Balance keyword vs semantic:
- Keyword heavy (0.7 keyword, 0.3 vector): Technical docs, exact terms matter
- Balanced (0.5, 0.5): General purpose
- Semantic heavy (0.3 keyword, 0.7 vector): Natural language queries

**Query Preprocessing:**
- Remove noise words
- Expand abbreviations
- Add context from conversation history

KEY POINTS:
- Precision vs recall tradeoff: threshold adjustments
- Hybrid search combines keyword + vector for best results
- Reranking improves quality at cost of latency
- Query expansion improves recall for varied terminology

EXAM TIP: 'Improve retrieval quality' → hybrid search, reranking, query expansion, metadata filtering

CONCEPT: RAG Architecture Patterns
================================================================================
Different RAG patterns for different use cases.

**Pattern 1: Basic RAG (Most Common)**
Query → Embed → Search → Top-K Chunks → FM → Response

Use Knowledge Bases RetrieveAndGenerate:
response = bedrock_agent.retrieve_and_generate(
    input={'text': user_query},
    retrieveAndGenerateConfiguration={
        'type': 'KNOWLEDGE_BASE',
        'knowledgeBaseConfiguration': {
            'knowledgeBaseId': 'KB_ID',
            'modelArn': 'arn:aws:bedrock:...:anthropic.claude-3-sonnet'
        }
    }
)

Best for: Simple Q&A, documentation search

**Pattern 2: Agentic RAG**
Query → Agent reasons → Decides to search → Retrieves → Reasons → May search again → Response

The agent decides when/what to search based on reasoning:
# Agent instructions
"""
You have access to a knowledge base about company policies.
Use it when users ask about policies, procedures, or guidelines.
You may need to search multiple times for complex questions.
"""

Best for: Complex questions requiring multiple lookups, conversation context

**Pattern 3: Multi-Index RAG**
Query → Route to appropriate index → Search specific KB → Generate

Different knowledge bases for different topics:
# Router (could be LLM or rules-based)
if 'policy' in query.lower():
    kb_id = 'POLICY_KB'
elif 'technical' in query.lower():
    kb_id = 'TECHNICAL_KB'
else:
    kb_id = 'GENERAL_KB'

Best for: Large organizations with distinct knowledge domains

**Pattern 4: Conversational RAG**
Query + History → Reformulate → Search → Generate with context

Incorporate conversation history:
def conversational_rag(query, history):
    # Reformulate query with context
    contextualized_query = llm.generate(
        f"Given this conversation: {history}\n"
        f"Reformulate this follow-up question as standalone: {query}"
    )

    # Search with contextualized query
    results = retrieve(contextualized_query)

    # Generate with full context
    return generate(query, results, history)

Best for: Chatbots, multi-turn conversations

**Pattern 5: Self-Query RAG**
Query → LLM extracts filters → Structured search → Results

Use LLM to understand query intent:
# LLM extracts: topic=returns, date_range=last_30_days
filters = llm.extract_filters(query)
results = search(query_embedding, filters=filters)

Best for: Complex queries with implicit filters

KEY POINTS:
- Basic RAG: Simple retrieve-and-generate, use Knowledge Bases
- Agentic RAG: Agent decides when/what to search
- Multi-Index: Route queries to domain-specific KBs
- Conversational: Reformulate queries with conversation context

EXAM TIP: Match RAG pattern to use case: basic for simple Q&A, agentic for complex reasoning, conversational for chatbots

CONCEPT: RAG Quality - Evaluation and Debugging
================================================================================
How to measure and improve RAG quality.

**The RAG Quality Triangle:**

        Retrieval Quality
            /         \
           /           \
     Generation ←→ Groundedness
       Quality

All three must be good for overall quality.

**Evaluating Retrieval:**
def evaluate_retrieval(queries, expected_docs):
    metrics = []
    for query, expected in zip(queries, expected_docs):
        retrieved = retrieve(query, k=5)

        # Precision: relevant retrieved / total retrieved
        relevant_retrieved = set(retrieved) & set(expected)
        precision = len(relevant_retrieved) / len(retrieved)

        # Recall: relevant retrieved / total relevant
        recall = len(relevant_retrieved) / len(expected)

        metrics.append({'precision': precision, 'recall': recall})

    return aggregate(metrics)

**Evaluating Groundedness:**
Is the answer supported by retrieved documents?

def check_groundedness(answer, retrieved_docs):
    prompt = f"""
    Answer: {answer}

    Supporting Documents:
    {retrieved_docs}

    For each claim in the answer, determine if it is:
    1. Supported by the documents
    2. Not supported (hallucinated)
    3. Partially supported

    Return JSON with claim-by-claim analysis.
    """
    return llm.evaluate(prompt)

**Evaluating Generation Quality:**
- Relevance: Does answer address the question?
- Completeness: Are all aspects covered?
- Conciseness: Is it appropriately brief?

**RAGAS Framework:**
Popular RAG evaluation framework:
- Faithfulness: Answer grounded in context
- Answer Relevancy: Answer addresses question
- Context Relevancy: Retrieved context is useful
- Context Recall: Needed info was retrieved

**Debugging RAG Issues:**

Problem: Wrong answers
1. Check retrieval: Were right docs found?
   → If no: chunking/embedding/query issue
   → If yes: Continue...

2. Check context: Is answer in retrieved docs?
   → If no: Retrieval precision issue
   → If yes: Continue...

3. Check generation: Did FM use context correctly?
   → If no: Prompt engineering issue

Problem: "I don't know" when answer exists
1. Check if doc is indexed
2. Check similarity score (maybe too low)
3. Check chunk boundaries (answer split across chunks)
4. Check query reformulation (query doesn't match doc language)

**Golden Dataset Testing:**
Maintain test queries with known answers:

golden_dataset = [
    {'query': 'What is the refund policy?', 'expected': 'Within 30 days...'},
    {'query': 'How do I reset password?', 'expected': 'Go to settings...'},
    # ... more examples
]

def regression_test():
    for item in golden_dataset:
        result = rag_pipeline(item['query'])
        score = evaluate(result, item['expected'])
        assert score > 0.8, f"Regression on: {item['query']}"

KEY POINTS:
- Evaluate retrieval (precision/recall), groundedness, and generation quality
- RAGAS framework provides structured RAG evaluation
- Debug systematically: retrieval → context → generation
- Golden datasets catch regressions

EXAM TIP: 'Evaluate RAG quality' → retrieval metrics + groundedness + golden datasets

1. RAG grounds FM responses in your documents - reduces hallucination
2. Chunking strategy is critical: match to document structure, include overlap

================================================================================
COMMON MISTAKES
================================================================================
- Wrong chunk size (too big = poor precision, too small = lost context)
- Different embedding models for queries vs documents
- Ignoring retrieval quality (assuming it 'just works')
- Not using hybrid search when keyword matching would help
- No golden dataset for regression testing
